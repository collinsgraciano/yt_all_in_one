## 第一章：环境准备

### 1.1 必需安装的软件

在你的 Windows 电脑上安装以下软件（按顺序）：

#### ① Docker Desktop（必须）

Docker 是整个开发环境的基础，它让你不用在本地装 Python、PostgreSQL 等一堆东西。

- 下载地址：https://www.docker.com/products/docker-desktop/
- 安装后启动 Docker Desktop，等待右下角鲸鱼图标变为稳定状态
- **验证安装**：打开命令提示符（CMD），输入：
  ```cmd
  docker --version
  docker-compose --version
  ```
  应显示版本号，如 `Docker version 24.x.x` 和 `Docker Compose version v2.x.x`

> **注意**：本项目仅使用 Python + PostgreSQL，不再需要 Redis、Node.js 等额外软件。前端使用 Jinja2 服务端渲染，无需前端构建步骤。

#### ② OpenSSH 客户端（远程部署需要）

Windows 10/11 通常已内置。验证：
```cmd
ssh -V
```
如果提示找不到命令，去「设置 → 应用 → 可选功能 → 添加功能」安装 OpenSSH 客户端。

### 1.2 项目文件确认

确保你的项目目录结构完整，关键文件如下：

```

│   ├── api/                    ← API 路由
```

## 第三章：日常开发工作流

这是你日常最常用的工作流。核心思想：**改完代码保存，服务自动热重载，不需要重建镜像**。

### 3.1 启动开发环境

每次开始 coding 前只需执行：

```cmd
scripts\dev.bat
```

> 因为首次已经构建过镜像，后续启动只需要几秒钟。

### 3.2 修改后端代码（自动热重载）

**场景**：你修改了 `backend/api/channels.py` 中的某个接口。

1. 用编辑器打开 `backend/api/channels.py`，修改代码并保存
2. 看运行 `dev.bat` 的终端窗口，会出现类似日志：
   ```
   WARNING:  WatchFiles detected changes in 'api/channels.py'
   INFO:     Reloading...
   INFO:     Application startup complete.
   ```
3. 刷新浏览器，修改已生效！

**无需任何手动重启操作。**

### 3.3 修改 Jinja2 模板

**场景**：你修改了 `backend/templates/dashboard.html` 中的页面布局。

1. 编辑 `backend/templates/dashboard.html`，保存修改
2. 刷新浏览器，修改已生效（模板文件修改后立即反映，无需重启）

### 3.4 修改 Pipeline 代码

**场景**：你修改了 `pipeline/pipeline.py` 中的处理逻辑。

1. 编辑 `pipeline/pipeline.py`，保存修改
2. 手动重启 Web 服务使新代码生效：

```cmd
REM 在运行 dev.bat 的终端按 Ctrl+C 停止，然后重新运行
scripts\dev.bat
```

> Pipeline 代码通过 Docker 卷挂载映射到容器内，重启 Web 容器即可加载新代码。

### 3.5 修改了 Python 依赖（需要重建镜像）

**场景**：你在 `requirements.txt` 中新增了一个包，比如 `httpx==0.27.0`。

此时热重载无法安装新依赖，需要重建镜像：

```cmd
scripts\rebuild-deps.bat
```

> 这个操作只在依赖变更时执行，日常改代码不需要。

### 3.6 各类修改的操作速查表

| 你修改了什么 | 需要做什么 | 耗时 |
|------------|----------|------|
| `backend/` 下的 `.py` 文件 | 什么都不用做，自动热重载 | 0 秒 |
| `backend/templates/` 下的 `.html` 文件 | 什么都不用做，刷新浏览器即可 | 0 秒 |
| `pipeline/` 下的 `.py` 文件 | 重启 Web 容器 | ~5 秒 |
| `requirements.txt` | 运行 `scripts\rebuild-deps.bat` | 2-5 分钟 |
| `docker/Dockerfile.web` | 运行 `scripts\rebuild-deps.bat` | 2-5 分钟 |
| `docker-compose.yml` 或 `docker-compose.dev.yml` | 停止后重新运行 `scripts\dev.bat` | ~10 秒 |

## 第四章：远程服务器部署 — 首次配置

### 4.1 服务器环境准备

在你的远程 Linux 服务器上安装 Docker：

```bash
# SSH 登录服务器
ssh root@你的服务器IP

# 安装 Docker（Ubuntu/Debian）
curl -fsSL https://get.docker.com | sh

# 验证安装
docker --version
docker-compose --version
```

### 4.2 验证 SSH 密码连接

在本地 Windows 电脑上测试 SSH 连接：

```cmd
ssh root@你的服务器IP
```

输入服务器密码，如果成功登录说明连接正常。输入 `exit` 退出。

> Windows 10 自带 OpenSSH 客户端，无需额外安装。部署脚本使用密码方式连接服务器。

### 4.3 初始化 Git 并推送到 GitHub

在本地项目根目录执行：

```cmd
:: 初始化 git 仓库
git init
git add -A
git commit -m "initial commit"
git branch -M main

:: 添加远程仓库（替换成你的地址）
git remote add origin https://github.com/YOUR_USER/audiobook-manager.git

:: 推送到 GitHub
git push -u origin main
```

> **认证说明**：GitHub 不支持密码推送。请使用 Personal Access Token：
> 1. 打开 https://github.com/settings/tokens → Generate new token (classic)
> 2. 勾选 `repo` 权限
> 3. 推送时用 token 代替密码
> 4. 或一键脚本：`scripts\git-deploy.bat "initial commit"`

### 4.4 服务器首次部署

SSH 登录服务器后：

```bash
# 安装 git
apt install -y git

# 克隆仓库
git clone https://github.com/YOUR_USER/audiobook-manager.git /opt/audiobook
cd /opt/audiobook

# 创建 .env 配置
cp .env.example .env
nano .env
```

执行部署：
```bash
bash scripts/git-server-deploy.sh
```

脚本会自动完成 git pull → 检查 .env → 构建镜像 → 启动服务 → 健康检查。
首次部署约 5-10 分钟（构建镜像）。

### 4.5 首次部署后：配置服务器环境变量

首次部署后，服务器的 `.env` 是从模板复制的，需要 SSH 登录修改：

```bash
ssh root@123.45.67.89
cd /opt/audiobook
nano .env
```

修改为：
```ini
DB_MODE=self
POSTGRES_PASSWORD=你的强密码
SECRET_KEY=你的随机密钥
BASE_URL=http://123.45.67.89:8080
```

> 如果使用外部数据库，改为：
> ```ini
> DB_MODE=external
> EXTERNAL_DATABASE_URL=postgresql://user:pass@your-db-host:5432/audiobook
> SECRET_KEY=你的随机密钥
> BASE_URL=http://123.45.67.89:8080
> ```

保存后重启服务：
```bash
bash scripts/deploy.sh
# 或手动：docker-compose -f docker-compose.yml -f docker-compose.self-db.yml up -d
```

### 4.6 验证部署

在浏览器中访问：
```
http://123.45.67.89:8080
```

如果看到登录页面，输入默认密码 `inriynisse` 后进入系统，部署成功！

## 第七章：脚本原理详解

### 7.1 开发环境为什么不用每次重建镜像？

核心原理是 Docker 的**卷挂载（Volume Mount）**。

生产环境（`docker-compose.yml`）中，代码是在构建镜像时 COPY 进去的：
```dockerfile
COPY backend/ /app/backend/
```

开发环境（`docker-compose.dev.yml`）中，代码通过卷挂载实时映射：
```yaml
volumes:
  - ./backend:/app/backend    # 本地 backend/ 目录映射到容器内 /app/backend/
```

这意味着你修改本地 `backend/` 下的文件，容器内同名文件立即变化。配合 `uvicorn --reload`，就能实现热重载。

### 7.2 热重载的工作原理

`docker-compose.dev.yml` 中 Web 服务的启动命令：
```yaml
command: >
  uvicorn backend.main:app
  --host 0.0.0.0
  --port 8080
  --reload              # ← 关键：启用热重载
  --reload-dir /app/backend  # ← 监听这个目录
```

`uvicorn --reload` 使用 `watchfiles` 库监听文件变化，检测到 `.py` 文件修改后自动重新加载应用。

### 7.3 任务执行机制

项目使用 Python `threading.Thread` 执行后台任务（替代 Celery + Redis）：

1. 用户在 Web 界面发起任务
2. `task_service.py` 在数据库 `run_tasks` 表中创建任务记录
3. 启动一个后台线程执行 Pipeline
4. Pipeline 运行过程中，日志通过 `log_interceptor.py` 写入 PostgreSQL `run_task_logs` 表
5. 前端通过 API 轮询 `/api/tasks/{task_id}/logs/recent` 获取最新日志

**停止任务**：在 `run_tasks` 表中设置 `stop_requested = true`，Pipeline 在下一个检查点检测到后优雅退出。

### 7.4 远程部署的智能构建策略

`git-server-deploy.sh` 通过 MD5 哈希对比判断是否需要重建镜像：

```bash
# 计算当前 requirements.txt 的哈希
CURRENT_HASH=$(md5sum requirements.txt | awk '{print $1}')

# 读取上次部署时记录的哈希
LAST_HASH=$(cat .last_requirements_hash)

# 对比
if [ "$CURRENT_HASH" != "$LAST_HASH" ]; then
    NEED_REBUILD=true  # 哈希不同，需要重建
fi
```

同理，Dockerfile 的内容也会被哈希对比。只有当依赖文件变化时才重建镜像，否则直接 `docker-compose up -d` 重启容器（秒级完成）。

### 7.5 .gitignore 排除策略

项目 `.gitignore` 确保以下内容不会被推送到 GitHub：

```
.git/            ← 已由 git 管理
__pycache__/     ← Python 缓存
.venv / venv     ← Python 虚拟环境
.env             ← 敏感配置，服务器有自己的 .env
*.log            ← 日志文件
tmp / .idea / .vscode ← 临时文件和 IDE 配置
data/ / output/  ← 生成的数据
```

推送到 GitHub 的只有核心代码和配置文件：`backend/`、`pipeline/`、`docker/`、`docker-compose.yml`、`requirements.txt`、`.env.example`。仓库体积小，clone 和 pull 都很快。

## 第八章：常见问题排查

### 问题 1：`dev.bat` 启动时报 "Docker 未运行"

**原因**：Docker Desktop 没有启动。

**解决**：启动 Docker Desktop，等待右下角鲸鱼图标变稳定后重试。

### 问题 2：Web 服务访问不了（8080 端口）

**排查步骤**：

```cmd
REM 检查容器是否在运行
docker-compose -f docker-compose.yml -f docker-compose.self-db.yml -f docker-compose.dev.yml ps

REM 查看 Web 服务日志
docker-compose -f docker-compose.yml -f docker-compose.self-db.yml -f docker-compose.dev.yml logs web

REM 检查端口是否被占用
netstat -ano | findstr :8080
```

**常见原因**：
- 8080 端口被其他程序占用 → 修改 `docker-compose.yml` 中的端口映射
- Web 容器启动失败 → 查看日志中的错误信息

### 问题 3：任务不执行

**排查步骤**：

```cmd
REM 查看 Web 日志（任务在 Web 容器的后台线程中执行）
docker-compose -f docker-compose.yml -f docker-compose.self-db.yml -f docker-compose.dev.yml logs web

REM 检查数据库中的任务状态
docker exec audiobook_postgres psql -U audiobook_app -d audiobook -c "SELECT task_id, channel_name, status FROM run_tasks ORDER BY created_at DESC LIMIT 10;"
```

**常见原因**：
- Pipeline 导入失败 → 查看 Web 日志中的 ImportError 信息
- 数据库连接失败 → 检查 `.env` 中的 `POSTGRES_PASSWORD`

### 问题 4：服务器 git pull 失败

**排查步骤**：

```bash
# SSH 登录服务器
cd /opt/audiobook

# 手动 git pull 查看错误
git pull

# 如果认证失败，配置 token 认证
git remote set-url origin https://USER:TOKEN@github.com/USER/REPO.git
git pull
```

**常见原因**：
1. 仓库为 Private，服务器没有配置认证
2. Token 已过期
3. 网络问题

### 问题 5：远程部署时镜像构建失败

**排查步骤**：SSH 登录服务器查看详细日志：

```bash
cd /opt/audiobook
docker-compose build 2>&1 | tee build.log
```

**常见原因**：
- 服务器网络慢，pip 下载超时 → 配置国内镜像源
- 磁盘空间不足 → `df -h` 检查，清理 `docker system prune`

### 问题 6：数据库连接失败

**排查步骤**：

```cmd
REM 检查 PostgreSQL 容器状态
docker-compose -f docker-compose.yml -f docker-compose.self-db.yml -f docker-compose.dev.yml ps postgres

REM 查看数据库日志
docker-compose -f docker-compose.yml -f docker-compose.self-db.yml -f docker-compose.dev.yml logs postgres
```

**常见原因**：
- `.env` 中 `POSTGRES_PASSWORD` 与首次创建时不一致
- 数据卷损坏 → 停止服务后删除卷重建：`docker-compose down -v` 然后 `dev.bat`（⚠️ 会丢失所有数据）

### 问题 7：如何重置整个开发环境

如果一切都很混乱，想从头开始：

```cmd
REM 停止并删除所有容器和卷（⚠️ 数据全丢）
docker-compose -f docker-compose.yml -f docker-compose.self-db.yml -f docker-compose.dev.yml down -v

REM 重新构建镜像
docker-compose -f docker-compose.yml -f docker-compose.self-db.yml -f docker-compose.dev.yml build

REM 启动
scripts\dev.bat
```
