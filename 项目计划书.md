## 1. 项目概述

### 1.1 项目目标

基于现有 `pipeline/` 有声书处理流水线（原 Colab Notebook 部署模式），开发一个 **Python Web 管理后台**，实现：

| 目标 | 说明 |
|------|------|
| **多频道统一管理** | 在一个 Web 界面中同时管理多个 YouTube 频道的配置、凭证和运行参数 |
| **频道认证增删** | 通过 Web 界面完成新频道的 YouTube OAuth 授权、凭证写入/删除，无需手动操作数据库 |
| **视频上传控制** | 在网页上对一个或多个频道发起视频处理+上传任务，实时查看进度和日志 |
| **数据库配置管理** | 在网页上可视化管理 PostgreSQL 中的所有配置表（频道设置、Token、书籍库等），支持增删改查和批量保存 |
| **Docker 一键部署** | 提供 `docker-compose up -d` 一键启动 Web 服务 + PostgreSQL |

### 1.2 与现有系统的关系

```
┌──────────────────────────────────────────────────────────┐
│                    现有 Colab 模式                        │
│  colab_loader.ipynb → 手动改参数 → 手动 OAuth → 手动运行   │
│  缺点: 每次都要手动改 Notebook、手动粘贴 OAuth URL、       │
│        不能同时管理多频道、无法远程控制                     │
└──────────────────────────────────────────────────────────┘
                           ↓ 升级为
┌──────────────────────────────────────────────────────────┐
│                    新 Web 管理后台模式                     │
│  浏览器访问 Web → 配置频道 → 一键 OAuth → 一键发起任务     │
│  Docker 部署 → 7×24 在线 → 多频道管理 → 实时监控          │
└──────────────────────────────────────────────────────────┘
```

### 1.3 核心价值

- **从手动到自动化**: 把 Colab Notebook 里的 7 个手动步骤全部变成 Web 界面操作
- **从单频道到多频道**: 一个后台管理 N 个 YouTube 频道，排队执行互不干扰
- **从临时运行到持久服务**: Docker 部署，7×24 小时在线，断线自动恢复
- **从黑盒到透明**: 实时日志轮询、任务进度可视化、历史记录可追溯
- **架构极简**: 仅 Python + PostgreSQL，无 Redis/Celery/消息队列，部署和维护成本低

## 2. 参考代码分析总结

### 2.1 现有 pipeline 架构

`pipeline/` 包含 13 个模块，分层组织：

```
Layer 0: config.py (95+ 配置项) + runtime.py (日志/工具)
Layer 1: db.py / state.py / audio.py / deepfilter.py / bgm.py / music_download.py / cover.py / seo.py
Layer 2: youtube.py (上传/播放列表/本地化)
Layer 3: podcast.py (monkey-patch 注入)
Layer 4: pipeline.py (run_pipeline 主流程编排)
```

### 2.2 现有数据库表（6 张）

| 表名 | 用途 | 主键 |
|------|------|------|
| `books` | 书籍元数据（book_id, book_name, category, book_data, status, tags） | book_id |
| `book_processing_states` | 断点续跑状态（book_id + project_flag） | (book_id, project_flag) |
| `youtube_credentials` | YouTube OAuth 令牌（channel_name → token_json） | channel_name |
| `modelscope_tokens` | AI 生图 Token | channel_name |
| `channel_runtime_settings` | 频道级运行配置（channel_name + setting_key → setting_value） | (channel_name, setting_key) |
| `task_queue` | 任务队列 | book_id |

### 2.3 现有 YouTube OAuth 流程

```
1. 用户提供 Google Cloud OAuth client_secret.json
2. Notebook 运行 InstalledAppFlow → 生成授权 URL
3. 用户手动打开 URL → 登录 → 复制 localhost 回调 URL
4. Notebook 解析 code → fetch_token → 写入 youtube_credentials 表
```

### 2.4 现有配置系统

- `config.py` 模块维护 95+ 个全局配置项
- `apply_runtime_config(dict)` 合并默认值与传入配置，写入模块全局
- `channel_runtime_settings` 表存储频道级配置（key-value 模式）
- 运行时通过 `apply_cloud_runtime_overrides()` 从数据库覆盖本地配置

### 2.5 现有运行流程

```
run_pipeline(config)
  → apply_runtime_config()
  → validate_runtime_config()
  → PostgreSQL 连通性检查
  → apply_cloud_runtime_overrides()  # 从 DB 读取云端配置
  → sync_music_library_if_enabled()  # 下载音乐库
  → 拉取 books 列表 + 断点恢复状态
  → for each book:
      → 下载章节音频 → 降噪 → BGM 混音 → 封面 → SEO → MP4 封装 → 上传 YouTube
  → 写入运行报告
```

### 2.6 复用策略

| 模块 | 复用方式 |
|------|----------|
| `pipeline/` 全包 | **原封不动复用**，作为 Web 后端的核心执行引擎 |
| `config.py` 配置系统 | **复用**，Web 后端通过 `apply_runtime_config()` 注入配置 |
| `db.py` 数据库操作 | **复用**，Web 后端直接调用其 CRUD 函数管理数据 |
| `youtube.py` OAuth | **适配**，把手动 URL 复制改为 Web 重定向流程 |
| `colab_loader.ipynb` | **废弃**，功能完全由 Web 后端替代 |

### 3.2 组件职责

| 组件 | 技术 | 职责 |
|------|------|------|
| **Web API 服务** | FastAPI + Uvicorn | REST API、Jinja2 页面渲染、OAuth 回调处理、后台线程任务执行 |
| **前端页面** | Jinja2 模板 + Bootstrap 5 | 频道管理界面、配置编辑器、任务监控面板、日志查看 |
| **任务执行** | Python `threading.Thread` + `threading.Lock` | 后台执行 `pipeline.run_pipeline()`，串行排队，避免模块级全局变量冲突 |
| **数据库** | PostgreSQL 16 | 频道配置、凭证、书籍库、处理状态、任务记录、OAuth State、任务日志 |
| **数据卷** | Docker Volume | 音频/视频/封面临时文件、音乐库、运行报告 |

### 3.3 多频道任务执行模型

```
                    ┌─────────────────┐
                    │  Web API 收到    │
                    │  "频道A+频道B     │
                    │   同时上传" 请求   │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │  任务调度器      │
                    │  (threading)     │
                    └───┬─────────┬───┘
                        │         │
              ┌─────────▼──┐  ┌──▼─────────┐
              │ Thread-A   │  │ Thread-B   │
              │ 频道A配置   │  │ 频道B配置   │
              │ 等待锁...   │  │ 等待锁...   │
              └──────┬─────┘  └────────────┘
                     │
                     ▼ 获取 _pipeline_lock
              ┌──────────────┐
              │ run_pipeline  │  ← 同一时间只有一个 pipeline 在运行
              │ (频道A)       │     pipeline 使用模块级全局变量，
              │ status=running│     非线程安全，必须串行
              └──────┬───────┘
                     │ 完成后释放锁
                     ▼
              ┌──────────────┐
              │ run_pipeline  │  ← Thread-B 获取锁后开始执行
              │ (频道B)       │
              └──────────────┘
```

每个频道在独立的后台线程中运行，通过 `threading.Lock` 确保同一时间只有一个 `pipeline.run_pipeline()` 在执行。任务在获取锁之前保持 `queued` 状态，获取锁后才转为 `running`。

> **设计决策**: Pipeline 模块使用模块级全局变量存储配置（非线程安全），因此不能真正并行执行。串行模型简单可靠，对于有声书处理的场景（任务耗时长但不需要高并发）完全够用。

async def create_task(request: Request):
    body = await request.json()
    channel_names = body["channel_names"]  # 支持多频道
    task_type = body.get("task_type", "full_pipeline")
    config_overrides = body.get("config_overrides", {})

    tasks = []
    for channel_name in channel_names:
        # 1. 检查频道是否已有运行中任务
        existing = check_running_task(channel_name)
        if existing:
            tasks.append({"task_id": existing["task_id"], "status": "already_running"})
            continue

        # 2. 读取频道配置并合并覆盖
        runtime_config = build_runtime_config(channel_name, config_overrides)

        # 3. 创建任务记录
        task_id = create_task_record(channel_name, task_type, runtime_config)

        # 4. 启动后台线程
        thread = threading.Thread(
            target=_run_pipeline_in_thread,
            args=(task_id, channel_name, runtime_config),
            daemon=True,
        )
        thread.start()

        tasks.append({"task_id": task_id, "channel_name": channel_name, "status": "queued"})

    return {"tasks": tasks}

#### 后台线程执行逻辑

```python
# task_service.py
_pipeline_lock = threading.Lock()  # 全局串行锁

def _run_pipeline_in_thread(task_id: str, channel_name: str, runtime_config: dict):
    """在后台线程中执行 pipeline。

    pipeline 使用模块级全局变量存储配置（非线程安全），
    因此通过 _pipeline_lock 串行执行，确保同一时间只有一个 pipeline 在运行。
    任务在获取锁之前保持 'queued' 状态，获取锁后才转为 'running'。
    """
    # 安装日志拦截器
    handler = install_log_interceptor(task_id)

    # 等待获取串行锁
    _pipeline_lock.acquire()

    try:
        # 获取锁后再次检查是否已被取消
        if get_task_status(task_id) == "cancelled":
            return

        # 标记任务为运行中
        update_task_status(task_id, "running")

        # 应用配置并执行
        apply_runtime_config(runtime_config)
        result = run_pipeline(runtime_config)

        # 检查是否因用户停止而返回
        if check_stop_flag(task_id):
            update_task_status(task_id, "cancelled", stop_reason="用户手动停止")
        else:
            update_task_status(task_id, "success", result_json=result)

    except Exception as e:
        if check_stop_flag(task_id):
            update_task_status(task_id, "cancelled", stop_reason="用户手动停止")
        else:
            update_task_status(task_id, "failed", error_msg=str(e))

    finally:
        uninstall_log_interceptor(task_id, handler)
        _pipeline_lock.release()
```

#### 日志拦截器

```python
# log_interceptor.py
import sys
import logging

class TaskLogInterceptor:
    """拦截 pipeline 的 stdout 输出和 logging 日志，写入 PostgreSQL。"""

    def __init__(self, task_id: str):
        self.task_id = task_id
        self.original_stdout = sys.stdout

    def write(self, text: str):
        # 保留原始输出
        self.original_stdout.write(text)
        # 写入数据库
        if text.strip():
            insert_task_log(self.task_id, "INFO", text.strip())

    def flush(self):
        self.original_stdout.flush()

def install_log_interceptor(task_id: str):
    """安装日志拦截器：替换 sys.stdout + 挂载 logging Handler。"""
    interceptor = TaskLogInterceptor(task_id)
    sys.stdout = interceptor

    handler = DatabaseLogHandler(task_id)
    handler.setLevel(logging.INFO)
    logging.getLogger().addHandler(handler)

    return handler

def uninstall_log_interceptor(task_id: str, handler):
    """卸载日志拦截器。"""
    sys.stdout = sys.__stdout__  # 恢复原始 stdout
    logging.getLogger().removeHandler(handler)
```

### 7.5 OAuth 状态管理

```python
# OAuth state 存储（防止 CSRF）
# 使用 PostgreSQL oauth_states 表替代 Redis
import secrets

def start_oauth(channel_name: str) -> dict:
    """发起 OAuth，返回授权 URL。"""
    # 1. 读取该频道的 client_secret
    client_secret = get_oauth_client_secret(channel_name)

    # 2. 生成 state 并存入数据库
    state = secrets.token_urlsafe(32)
    execute(
        sql.SQL("INSERT INTO public.oauth_states (state, channel_name) VALUES (%s, %s)"),
        (state, channel_name),
    )

    # 3. 顺手清理过期 state（超过 TTL 的记录）
    execute(
        sql.SQL("DELETE FROM public.oauth_states WHERE created_at < now() - make_interval(secs => %s)"),
        (app_settings.oauth_state_ttl_seconds,),  # 默认 600 秒
    )

    # 4. 构建 Flow
    flow = Flow.from_client_config(
        client_secret,
        scopes=["https://www.googleapis.com/auth/youtube"],
        state=state,
        redirect_uri=f"{settings.BASE_URL}/api/oauth/callback",
    )
    auth_url, _ = flow.authorization_url(prompt="consent", access_type="offline")

    return {"auth_url": auth_url, "state": state}

def handle_oauth_callback(code: str, state: str) -> str:
    """OAuth 回调处理。"""
    # 1. 从数据库验证 state
    row = fetch_one(
        sql.SQL("SELECT channel_name FROM public.oauth_states WHERE state = %s"),
        (state,),
    )
    if not row:
        raise ValueError("无效或过期的 state，请重新发起授权")

    channel_name = row["channel_name"]

    # 2. 删除已使用的 state
    execute(
        sql.SQL("DELETE FROM public.oauth_states WHERE state = %s"),
        (state,),
    )

    # 3. 读取 client_secret 并构建 Flow
    client_secret = get_oauth_client_secret(channel_name)
    flow = Flow.from_client_config(
        client_secret,
        scopes=["https://www.googleapis.com/auth/youtube"],
        state=state,
        redirect_uri=f"{settings.BASE_URL}/api/oauth/callback",
    )
    flow.fetch_token(code=code)
    creds = flow.credentials
    token_dict = json.loads(creds.to_json())

    # 4. 写入 youtube_credentials（UPSERT）
    execute(
        sql.SQL("""
            INSERT INTO public.youtube_credentials (channel_name, token_json, updated_at)
            VALUES (%s, %s, now())
            ON CONFLICT (channel_name)
            DO UPDATE SET token_json = EXCLUDED.token_json, updated_at = EXCLUDED.updated_at
        """),
        (channel_name, Jsonb(token_dict)),
    )

    # 5. 更新频道状态
    update_oauth_status(channel_name, "authorized")

    return channel_name
```

### 8.3 串行控制

```python
# 全局串行锁 — pipeline 使用模块级全局变量（非线程安全），必须串行执行
_pipeline_lock = threading.Lock()

def _run_pipeline_in_thread(task_id, channel_name, runtime_config):
    # 任务在获取锁之前保持 'queued' 状态
    _pipeline_lock.acquire()
    try:
        # 获取锁后才转为 'running'
        update_task_status(task_id, "running")
        # ... 执行 pipeline ...
    finally:
        _pipeline_lock.release()
```

> **为什么串行？** Pipeline 的 `config.py` 使用模块级全局变量存储配置（如 `DOWNLOAD_WORKERS`、`MAX_RUNTIME_HOURS` 等），多线程同时调用 `apply_runtime_config()` 会互相覆盖。串行执行是最简单可靠的解决方案。

### 8.4 任务停止机制

```python
# 通过数据库 stop_requested 标志位实现优雅退出
def stop_task(task_id: str) -> dict:
    """停止运行中的任务。"""
    # 1. 设置数据库停止标志
    execute(
        sql.SQL("UPDATE public.run_tasks SET stop_requested = true WHERE task_id = %s"),
        (task_id,),
    )

    # 2. 更新状态为 cancelled（仅在任务仍处于 queued/running 时）
    execute(
        sql.SQL("UPDATE public.run_tasks SET status = 'cancelled', stop_reason = '用户手动停止', "
                "finished_at = now() WHERE task_id = %s AND status IN ('queued', 'running')"),
        (task_id,),
    )
    return {"task_id": task_id, "status": "cancelled", "message": "任务已停止"}
```

Pipeline 内部通过 `_check_db_stop_flag()` 函数定期查询 `run_tasks.stop_requested` 列，检测到 `true` 后优雅退出当前处理循环。

### 8.5 日志获取（API 轮询）

前端通过定时轮询 API 获取最新日志，无需 WebSocket：

```python
# GET /api/tasks/{task_id}/logs/recent?after_id=123
async def get_recent_logs(task_id: str, after_id: int = 0):
    """获取指定 ID 之后的最新日志。"""
    rows = fetch_all(
        sql.SQL("""
            SELECT id, log_level, message, created_at
            FROM public.run_task_logs
            WHERE task_id = %s AND id > %s
            ORDER BY id ASC
            LIMIT 200
        """),
        (task_id, after_id),
    )
    return {"logs": rows, "last_id": rows[-1]["id"] if rows else after_id}
```

前端 JavaScript 每 3 秒轮询一次，传入上一次获取的最大 `id`，实现增量日志加载。

## 总结

本项目计划书基于对 `pipeline/` 包的深度分析，设计了一个完整的 Web 管理后台方案：

1. **最大化复用**: `pipeline/` 包原封不动作为执行引擎，Web 层只负责管理和调度
2. **数据库兼容**: 完全保留原有 6 张表结构，新增 6 张 Web 管理层表
3. **OAuth 现代化**: 从手动复制 URL 升级为 Web 自动重定向流程，State 存于 PostgreSQL
4. **串行任务执行**: 通过 `threading.Thread` + `threading.Lock` 实现任务排队和串行执行，简单可靠
5. **实时可观测**: 日志通过 stdout 拦截写入 PostgreSQL，前端 API 轮询获取
6. **架构极简**: 仅 Python + PostgreSQL 两个组件，无 Redis/Celery/前端构建步骤
7. **一键部署**: `docker-compose up -d` 启动全部服务

预计开发周期：**7-11 周**（Phase 1-5），可根据优先级裁剪范围。
