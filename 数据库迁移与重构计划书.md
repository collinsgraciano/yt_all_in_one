## 1. 项目现状分析

### 1.1 audiobook_pipeline 项目（源项目）

**定位**: 独立的有声书处理管线，负责从远程 PG 迁移数据 → DeepFilter 降噪 → 多 Bot 轮换上传 Telegram。

**数据库结构特点**:
- `books` 表与远程数据库 (85.121.48.55) **完全一致**（11列），仅新增 `book_status` 扩展列
- `audiobook_chapters` 表包含完整的 Worker 认领机制字段（`worker_id`, `claimed_at`, `error_message`）
- 所有分类/作者/标签都是**真实顶层列**，无需从 JSON 提取

**核心功能模块**:

| 模块 | 文件 | 功能 |
|------|------|------|
| 数据迁移 | `migrate.py` | 远程PG→本地PG，迁移books + 解析章节到audiobook_chapters |
| Worker核心 | `worker.py` | 认领pending章节→下载→降噪→多Bot上传→更新DB |
| 定时清理 | `cleanup.py` | 重置卡住的processing章节（用worker_id/claimed_at） |
| Web管理 | `web_admin/app.py` | Flask面板，搜索/编辑/分类筛选/批量重置 |
| VPS调度 | `vps_scheduler/scheduler.py` | 轮询PG + 触发HF Space Worker |
| HF Space | `hf_space/worker.py` | Serverless Worker部署 |

### 1.2 当前项目（目标项目）

**定位**: YouTube 有声书自动化管线，整合了书籍管理、YouTube上传、频道配置、TG音频缓存等完整功能。

**数据库结构特点**:
- `books` 表缺少 `book_status` 列（11列）
- `audiobook_chapters` 表缺少 `worker_id`, `claimed_at`, `error_message` 三列（11列）

**核心功能模块**:

| 模块 | 文件 | 功能 |

| 前端页面 | `backend/templates/books.html` | 书籍管理页面 |

| TG音频缓存 | `pipeline/tg_audio.py` | 查询audiobook_chapters + 多Bot下载 |

| 迁移脚本 | `migrate-tg-chapters/` | 旧格式books/chapters迁移工具 |

## 3. 迁移映射关系

### 3.1 数据迁移策略

**方案**: **完全删除**当前项目的 `books` 和 `audiobook_chapters` 两张表，然后从 `audiobook_pipeline` 项目**完整复制**表结构和数据过来。

```
audiobook_pipeline 的 PG (或远程源库 85.121.48.55)
  └── books 表 (12列, 含 book_status)
  └── audiobook_chapters 表 (14列, 含 worker_id/claimed_at/error_message)
        │
        ▼  完全替换 (DROP + CREATE + COPY)
当前项目的 PG
  └── books 表 (完全替换为 audiobook_pipeline 的结构 + 数据)
  └── audiobook_chapters 表 (完全替换为 audiobook_pipeline 的结构 + 数据)
  └── 其他表保持不变 (channels, run_tasks, global_settings 等)
```

**核心原则**: 不做增量 ALTER，直接 DROP 旧表 → 用 audiobook_pipeline 的表定义重建 → 导入数据。这样表结构干净一致，不残留旧字段和旧索引。

### 3.3 语义变更说明

| 概念 | 旧逻辑 (当前项目) | 新逻辑 (迁移后) |
|------|------------------|----------------|
| **书籍处理状态 (`status`)** | `status` 列混用 (远程状态 + 项目标记 flag) | **保持不变** — `status` 继续承担远程状态 + 项目 flag（防频道重复处理上传），`finalize_successful_book_for_project` 继续往 `status` 追加 flag |
| **章节完成标记 (`book_status`)** | ❌ 不存在 | **新增列** — 仅 `audiobook_pipeline` 自身标记某书所有章节是否上传完成 (pending/success)，迁移时保留数据，当前项目 pipeline **不主动改用**此列 |
| **分类提取** | `_top_level.category` JSON hack + COALESCE 三层回退 | 直接用 `category` 顶层列 |
| **章节认领** | 无认领机制 (pipeline直接处理) | `worker_id` + `claimed_at` 原子认领 |
| **错误追踪** | 无错误记录 | `error_message` 记录失败原因 |
| **卡住清理** | 无 (依赖pipeline自身重试) | `cleanup.py` 定时重置 processing 章节 |

## 4. 代码重构清单

### 4.1 数据库 Schema 修改

#### `docker/init-db.sql` — 完全替换 books 表定义

删除当前 books 表的 CREATE 语句，替换为 audiobook_pipeline 的版本（加 `public.` schema 前缀）：

```sql
-- 1. books — 书籍库 (完全替换为 audiobook_pipeline 结构)
DROP TABLE IF EXISTS public.books CASCADE;
CREATE TABLE public.books (
    book_id          text        PRIMARY KEY,
    book_name        text,
    author           text,
    category         text,
    total_chapters   integer,
    book_data        jsonb,
    tags             text[],
    note             text,
    status           text        DEFAULT '',
    created_at       timestamptz DEFAULT now(),
    updated_at       timestamptz DEFAULT now(),
    book_status      varchar(50) DEFAULT 'pending'  -- ← 新增: audiobook_pipeline章节完成标记(pending/success)
);

CREATE INDEX idx_books_category    ON public.books(category);
CREATE INDEX idx_books_status      ON public.books(status);
CREATE INDEX idx_books_book_status ON public.books(book_status);
CREATE INDEX idx_books_tags_gin    ON public.books USING gin(tags);
CREATE INDEX idx_books_updated_at  ON public.books(updated_at DESC);
```

#### `docker/init-db.sql` — 完全替换 audiobook_chapters 表定义

删除当前 audiobook_chapters 表的 CREATE 语句，替换为 audiobook_pipeline 的版本（加 `public.` schema 前缀，保留 pipeline 依赖的 `idx_audiobook_chapters_audio_url` 索引）：

```sql
-- 13. audiobook_chapters — 章节级 TG 缓存 (完全替换为 audiobook_pipeline 结构)
DROP TABLE IF EXISTS public.audiobook_chapters CASCADE;
CREATE TABLE public.audiobook_chapters (
    book_id              VARCHAR(255),
    chapter_id           VARCHAR(255),
    book_name            TEXT,
    chapter_name         TEXT,
    audio_url            TEXT,
    telegram_file_id     TEXT,
    telegram_message_id  BIGINT,
    telegram_bot_id      INT,
    telegram_bot_user_id BIGINT,
    upload_status        VARCHAR(50) DEFAULT 'pending',
    worker_id            VARCHAR(100),   -- ← 新增: Worker 认领机制
    claimed_at           TIMESTAMP,      -- ← 新增: 认领时间
    uploaded_at          TIMESTAMP,
    error_message        TEXT,           -- ← 新增: 错误追踪
    PRIMARY KEY (book_id, chapter_id)
);

-- audiobook_pipeline 的索引
CREATE INDEX idx_chapters_upload_status ON public.audiobook_chapters(upload_status);
CREATE INDEX idx_chapters_book_id       ON public.audiobook_chapters(book_id);
CREATE INDEX idx_chapters_book_status   ON public.audiobook_chapters(book_id, upload_status);

-- 保留当前项目 pipeline/tg_audio.py 依赖的索引 (按 audio_url 匹配章节)
CREATE INDEX idx_audiobook_chapters_audio_url ON public.audiobook_chapters(book_id, audio_url);
```

> **注意**: `docker/init-db.sql` 仅在数据库首次初始化时执行。已有数据库需要手动执行 DROP + CREATE + 导入数据（见第 5 节）。

-- ============================================================
-- 2. 重建 books 表 (audiobook_pipeline 结构)

    book_data        jsonb,

    note             text,

    updated_at       timestamptz DEFAULT now(),

);

CREATE INDEX idx_books_category    ON public.books(category);
CREATE INDEX idx_books_status      ON public.books(status);
CREATE INDEX idx_books_book_status ON public.books(book_status);
CREATE INDEX idx_books_tags_gin    ON public.books USING gin(tags);
CREATE INDEX idx_books_updated_at  ON public.books(updated_at DESC);

-- ============================================================
-- 3. 重建 audiobook_chapters 表 (audiobook_pipeline 结构)
-- ============================================================
CREATE TABLE public.audiobook_chapters (
    book_id              VARCHAR(255),
    chapter_id           VARCHAR(255),
    book_name            TEXT,
    chapter_name         TEXT,
    audio_url            TEXT,
    telegram_file_id     TEXT,
    telegram_message_id  BIGINT,
    telegram_bot_id      INT,
    telegram_bot_user_id BIGINT,
    upload_status        VARCHAR(50) DEFAULT 'pending',
    worker_id            VARCHAR(100),
    claimed_at           TIMESTAMP,

CREATE INDEX idx_chapters_book_id       ON public.audiobook_chapters(book_id);

EOF

## 7. 风险评估与回滚方案

### 7.1 风险评估

| 风险项 | 概率 | 影响 | 缓解措施 |
|--------|------|------|---------|
| 数据迁移丢失 TG 缓存 | 低 | 高 | 迁移前备份 `telegram_file_id` 数据 |
| `_top_level` 逻辑删除后分类丢失 | 中 | 中 | 迁移后验证 `category` 列覆盖率 |
| `status` 列被误改为 `book_status` | 低 | 高 | **明确不改** `status` 逻辑，`book_status` 仅作迁移数据保留 |
| pipeline 状态更新失败 | 低 | 高 | 代码中 try/except + 日志告警 |
| 迁移脚本字段不匹配 | 中 | 中 | 先 dry-run 预览 |

### 7.2 回滚方案

```bash
# 1. 停止服务
docker compose down

# 2. 恢复数据库备份

docker compose up -d
```
