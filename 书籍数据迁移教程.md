# 书籍数据迁移教程：从「下载掌阅有声书到tg」项目迁移到当前项目

> **迁移目标**：将旧项目 `下载掌阅有声书到tg` 中 `books` 表的数据，完整迁移到当前项目 `yt_aduio_book_one_to_all` 的 `books` 表中，并兼容新表结构。

---

## 1. 两个项目的 books 表结构对比

### 1.1 旧项目（源）books 表

旧项目的 `books` 表只有 **3 个字段**，书名、作者等信息全部嵌在 `book_data` JSONB 中：

```sql
-- 来源：下载掌阅有声书到tg/init.sql
CREATE TABLE books (
    book_id      VARCHAR(255) PRIMARY KEY,   -- 书籍 ID
    book_data    JSONB,                        -- 完整原始 JSON 数据
    book_status  VARCHAR(50) DEFAULT 'pending' -- 书级处理标记: pending/success
);
```

`book_data` JSON 内部结构（掌阅原始数据）：

| JSON 字段 | 说明 |
|---|---|
| `bookName` | 书名 |
| `bookAuthor` | 作者 |
| `bookDescription` | 简介（可能很长） |
| `tingChapterList` | 章节列表（每项含 `tingChapterId`, `chapterName`, `playUrl` 等） |
| `bookInfo` | 嵌套对象，部分版本的掌阅数据将章节放在此处 |

### 1.2 新项目（目标）books 表

新项目的 `books` 表有 **11 个字段**，将关键字段从 JSON 中提取为独立列：

```sql
-- 来源：docker/init-db.sql
CREATE TABLE public.books (
    book_id         text PRIMARY KEY,           -- 书籍 ID
    book_name       text,                        -- 书名（从 book_data.bookName 提取）
    author          text,                        -- 作者（从 book_data.bookAuthor 提取）
    category        text,                        -- 分类
    total_chapters  integer,                     -- 章节总数（从 tingChapterList 长度提取）
    book_data       jsonb,                       -- 完整原始 JSON 数据（保留）
    tags            text[],                      -- 标签数组
    note            text,                        -- 备注
    status          text DEFAULT '',             -- 状态（对应旧表的 book_status）
    created_at      timestamptz DEFAULT now(),    -- 创建时间
    updated_at      timestamptz DEFAULT now()     -- 更新时间
);
```

### 1.3 字段映射方案

| 旧表字段 | → | 新表字段 | 转换逻辑 |
|---|---|---|---|
| `book_id` | → | `book_id` | 直接映射，类型从 VARCHAR(255) → text |
| `book_data` | → | `book_data` | 直接映射（保留完整 JSON） |
| `book_data->>'bookName'` | → | `book_name` | 从 JSON 提取为独立列 |
| `book_data->>'bookAuthor'` | → | `author` | 从 JSON 提取为独立列 |
| — | → | `category` | 旧表无此字段，默认 NULL |
| `jsonb_array_length(book_data->'tingChapterList')` | → | `total_chapters` | 从章节列表长度提取 |
| — | → | `tags` | 旧表无此字段，默认 `'{}'` |
| — | → | `note` | 旧表无此字段，默认 NULL |
| `book_status` | → | `status` | `pending` → `pending`，`success` → `success` |
| — | → | `created_at` | 默认 `now()` |
| — | → | `updated_at` | 默认 `now()` |

### 1.4 旧项目额外表说明

旧项目还有一张 `audiobook_chapters` 表（章节级状态表），当前项目中**没有对应表**。当前项目使用 `book_processing_states` 表替代，通过 `project_flag`（频道名）关联。因此 `audiobook_chapters` 的数据**不在本次迁移范围内**。

## 3. 迁移方式选择

提供 **两种迁移方案**，按实际情况选择：

| 方案 | 适用场景 | 复杂度 |
|---|---|---|
| **方案 A：纯 SQL 迁移** | 旧库和新库在同一个 PostgreSQL 实例或可互相访问 | 低 |
| **方案 B：Python 脚本迁移** | 旧库和新库在不同服务器/不同 Docker 网络 | 中 |

---

## 4. 方案 A：纯 SQL 迁移（推荐）

> 适用条件：旧库和新库可以通过 `pg_dump` / `psql` 或 `dblink` 互相访问。

### 4.1 步骤 1：从旧库导出数据

```bash
# 设置旧库密码
export PGPASSWORD=inriynisse1991

# 导出 books 表数据（仅数据，不含表结构）
pg_dump -h 127.0.0.1 -p 5432 -U audiobook_app -d audiobook \
    --data-only --column-inserts --no-owner --no-privileges \
    --table=books \
    > /tmp/old_books_dump.sql
```

### 4.2 步骤 2：导入到新库（带字段转换）

由于旧表只有 3 列、新表有 11 列，直接导入会报错。需要通过临时表中转：

```sql
-- 在新库中执行

-- 1. 创建临时表（与旧表结构一致）
CREATE TEMP TABLE old_books_import (
    book_id     VARCHAR(255) PRIMARY KEY,
    book_data   JSONB,
    book_status VARCHAR(50) DEFAULT 'pending'
);

-- 2. 导入旧数据到临时表
-- 使用 psql 命令导入 dump 文件到临时表
\i /tmp/old_books_dump.sql
-- 注意：dump 文件中的 INSERT 语句会自动插入到 old_books_import 表
-- 因为 pg_dump --table=books 生成的 SQL 使用的是表名 "books"
-- 所以需要先修改 dump 文件中的表名，或者用以下方式：

-- 替代方案：直接用 COPY 导出 CSV 再导入
-- 在旧库执行:
-- \COPY (SELECT book_id, book_data, book_status FROM books) TO '/tmp/books_export.csv' WITH CSV;
-- 在新库执行:
-- \COPY old_books_import FROM '/tmp/books_export.csv' WITH CSV;
```

**更简单的替代方法**（推荐）——直接在新库通过 `dblink` 或 `postgres_fdw` 跨库查询：

```sql
-- 在新库中执行

-- 1. 安装 fdw 扩展
CREATE EXTENSION IF NOT EXISTS postgres_fdw;

-- 2. 创建外部服务器（指向旧库）
CREATE SERVER old_audiobook_db
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host '127.0.0.1', port '5432', dbname 'audiobook');

-- 3. 创建用户映射
CREATE USER MAPPING FOR audiobook_app
    SERVER old_audiobook_db
    OPTIONS (user 'audiobook_app', password 'inriynisse1991');

-- 4. 导入旧表定义
IMPORT FOREIGN SCHEMA public
    LIMIT TO (books)
    FROM SERVER old_audiobook_db
    INTO public;

-- 5. 重命名导入的外部表（避免与本地 books 表冲突）
ALTER TABLE public.books RENAME TO books_new_local;
-- 注意：上面这步会重命名本地 books 表！不要这样做。
-- 正确做法：先导入到不同 schema

-- ====== 推荐做法 ======

-- 1. 创建专用 schema 放旧表
CREATE SCHEMA IF NOT EXISTS old_import;

-- 2. 导入旧表到专用 schema
IMPORT FOREIGN SCHEMA public
    LIMIT TO (books)
    FROM SERVER old_audiobook_db
    INTO old_import;

-- 3. 执行迁移 SQL（从旧表提取字段到新表）
INSERT INTO public.books (book_id, book_name, author, category, total_chapters, book_data, tags, note, status, created_at, updated_at)
SELECT
    ob.book_id,
    -- 书名：优先从 bookName 提取，兼容 title/name
    COALESCE(
        ob.book_data->>'bookName',
        ob.book_data->>'title',
        ob.book_data->>'name',
        '未知_' || ob.book_id
    ) AS book_name,
    -- 作者
    ob.book_data->>'bookAuthor' AS author,
    -- 分类（旧表无此字段）
    NULL::text AS category,
    -- 章节总数：从 tingChapterList 列表长度提取，兼容多种字段名
    COALESCE(
        jsonb_array_length(ob.book_data->'tingChapterList'),
        jsonb_array_length(ob.book_data->'chapterList'),
        jsonb_array_length(ob.book_data->'chapters'),
        jsonb_array_length(ob.book_data->'bookInfo'->'tingChapterList'),
        0
    ) AS total_chapters,
    -- 完整 JSON 数据保留
    ob.book_data,
    -- 标签默认空数组
    ARRAY[]::text[] AS tags,
    -- 备注默认空
    NULL::text AS note,
    -- 状态映射：book_status → status
    COALESCE(ob.book_status, 'pending') AS status,
    now(),
    now()
FROM old_import.books ob
ON CONFLICT (book_id) DO UPDATE SET
    book_name      = EXCLUDED.book_name,
    author         = EXCLUDED.author,
    total_chapters = EXCLUDED.total_chapters,
    book_data      = EXCLUDED.book_data,
    status         = EXCLUDED.status,
    updated_at     = now()
WHERE public.books.book_id IS NULL  -- 只更新新库中不存在的记录（如需覆盖则去掉此行）
   OR public.books.book_data IS NULL; -- 或新库中 book_data 为空的记录

-- 4. 清理：删除外部表和 schema
DROP SCHEMA old_import CASCADE;
DROP SERVER old_audiobook_db;
```

### 4.3 步骤 3：验证迁移结果

```sql
-- 在新库中执行
SELECT
    COUNT(*) AS total,
    COUNT(*) FILTER (WHERE status = 'pending') AS pending,
    COUNT(*) FILTER (WHERE status = 'success') AS success,
    COUNT(*) FILTER (WHERE book_name IS NOT NULL) AS has_name,
    COUNT(*) FILTER (WHERE author IS NOT NULL) AS has_author,
    COUNT(*) FILTER (WHERE total_chapters > 0) AS has_chapters
FROM public.books;
```

## 5. 方案 B：Python 脚本迁移

> 适用条件：旧库和新库在不同服务器，或无法使用 `postgres_fdw`。

### 5.1 迁移脚本

项目已提供迁移脚本 `scripts/migrate_books_from_old_project.py`，可直接使用。

**功能**：
- 从旧库 `books` 表读取全部数据
- 解析 `book_data` JSON，提取书名、作者、章节数
- 批量写入新库 `public.books` 表
- 支持 `ON CONFLICT` 去重（已有记录跳过或更新）

使用方法：

```bash
# 基本用法（使用默认配置）
python scripts/migrate_books_from_old_project.py

# 指定旧库和新库连接串
python scripts/migrate_books_from_old_project.py \
    --source-dsn "postgresql://audiobook_app:inriynisse1991@127.0.0.1:5432/audiobook" \
    --target-dsn "postgresql://audiobook_app:your_password@127.0.0.1:5432/audiobook"

# 仅导入不存在的记录（不覆盖已有数据）
python scripts/migrate_books_from_old_project.py --skip-existing

# 覆盖已有记录（更新 book_data 等字段）
python scripts/migrate_books_from_old_project.py --overwrite

# 指定批量大小
python scripts/migrate_books_from_old_project.py --batch-size 500
```

### 5.2 迁移脚本核心逻辑

```python
# 简化版核心逻辑（完整脚本见 scripts/migrate_books_from_old_project.py）

import psycopg
from psycopg.types.json import Jsonb

# 从旧库读取
with psycopg.connect(source_dsn) as src_conn:
    with src_conn.cursor() as cur:
        cur.execute("SELECT book_id, book_data, book_status FROM books")
        rows = cur.fetchall()

# 写入新库
with psycopg.connect(target_dsn) as dst_conn:
    with dst_conn.cursor() as cur:
        for book_id, book_data_raw, book_status in rows:
            book_data = book_data_raw  # 已经是 dict（psycopg3 自动解析 jsonb）

            # 提取字段
            book_name = book_data.get('bookName') or book_data.get('title') or f'未知_{book_id}'
            author = book_data.get('bookAuthor', '')
            chapters = extract_chapters(book_data)  # 兼容多种字段名
            total_chapters = len(chapters) if chapters else 0

            # 插入
            cur.execute("""
                INSERT INTO public.books (book_id, book_name, author, total_chapters, book_data, tags, status)
                VALUES (%s, %s, %s, %s, %s, %s, %s)
                ON CONFLICT (book_id) DO NOTHING
            """, (book_id, book_name, author, total_chapters,
                  Jsonb(book_data), [], book_status or 'pending'))
```

如果发现有书名未正确提取的记录，可以手动修复：

```sql
-- 批量修复：从 book_data JSON 中重新提取 book_name
UPDATE public.books
SET book_name = book_data->>'bookName',
    author = book_data->>'bookAuthor',
    updated_at = now()
WHERE book_name IS NULL OR book_name = '';

-- 批量修复：重新计算 total_chapters
UPDATE public.books

    jsonb_array_length(book_data->'tingChapterList'),
    jsonb_array_length(book_data->'chapterList'),
    jsonb_array_length(book_data->'chapters'),
    0
),
    updated_at = now()
WHERE total_chapters IS NULL OR total_chapters = 0;
```

## 7. Docker 环境下的快速迁移（一体化命令）

如果两个项目的 PostgreSQL 都在同一台服务器的 Docker 中运行：

```bash
#!/bin/bash
set -euo pipefail

# ── 旧库连接信息 ──
OLD_CONTAINER="audiobook_pg"        # 旧项目容器名
OLD_USER="audiobook_app"
OLD_DB="audiobook"

# ── 新库连接信息 ──
NEW_CONTAINER="audiobook_postgres"   # 新项目容器名（根据实际修改）
NEW_USER="audiobook_app"
NEW_DB="audiobook"

echo ">>> 1. 从旧库导出 books 数据..."
docker exec "$OLD_CONTAINER" psql -U "$OLD_USER" -d "$OLD_DB" -c "
    COPY (SELECT book_id, book_data, book_status FROM books) TO '/tmp/books_export.csv' WITH CSV;
"

echo ">>> 2. 复制导出文件到宿主机..."
docker cp "$OLD_CONTAINER:/tmp/books_export.csv" /tmp/books_export.csv

echo ">>> 3. 复制到新库容器..."
docker cp /tmp/books_export.csv "$NEW_CONTAINER:/tmp/books_export.csv"

echo ">>> 4. 在新库中导入并转换..."
docker exec "$NEW_CONTAINER" psql -U "$NEW_USER" -d "$NEW_DB" <<'SQL'
-- 创建临时表
CREATE TEMP TABLE old_books_import (
    book_id     text,
    book_data   jsonb,
    book_status text DEFAULT 'pending'
);

-- 导入 CSV
COPY old_books_import FROM '/tmp/books_export.csv' WITH CSV;

-- 转换并插入到目标表
INSERT INTO public.books (book_id, book_name, author, category, total_chapters, book_data, tags, note, status, created_at, updated_at)
SELECT
    ob.book_id,
    COALESCE(
        ob.book_data->>'bookName',
        ob.book_data->>'title',
        ob.book_data->>'name',
        '未知_' || ob.book_id
    ),
    ob.book_data->>'bookAuthor',
    NULL,
    COALESCE(
        jsonb_array_length(ob.book_data->'tingChapterList'),
        jsonb_array_length(ob.book_data->'chapterList'),
        jsonb_array_length(ob.book_data->'chapters'),
        0
    ),
    ob.book_data,
    ARRAY[]::text[],
    NULL,
    COALESCE(ob.book_status, 'pending'),
    now(),
    now()
FROM old_books_import ob
ON CONFLICT (book_id) DO UPDATE SET
    book_name      = EXCLUDED.book_name,
    author         = EXCLUDED.author,
    total_chapters = EXCLUDED.total_chapters,
    book_data      = EXCLUDED.book_data,
    status         = EXCLUDED.status,
    updated_at     = now();

-- 验证
SELECT COUNT(*) AS migrated_count FROM public.books;
SQL

echo ">>> 5. 清理临时文件..."
rm -f /tmp/books_export.csv

echo "✅ 迁移完成"
```

## 8. 常见问题排查

### Q1: `jsonb_array_length` 报错 "cannot get array length of a non-array"

旧数据中可能有部分记录的 `tingChapterList` 字段不存在或为 null。使用 `COALESCE` 处理：

```sql
COALESCE(
    CASE
        WHEN jsonb_typeof(book_data->'tingChapterList') = 'array'
        THEN jsonb_array_length(book_data->'tingChapterList')
        ELSE 0
    END,
    0
)
```

### Q2: psycopg3 报错 "cannot adapt type dict"

在 Python 脚本中需要用 `Jsonb()` 包装 dict 类型：

```python
from psycopg.types.json import Jsonb
cur.execute("INSERT INTO books (book_data) VALUES (%s)", (Jsonb(book_data_dict),))
```

### Q3: 旧库的 `book_status` 值和新库 `status` 值不匹配

旧库使用 `pending` / `success`，新库 `status` 字段允许任意文本。直接映射即可：

```sql
-- 旧库 status 值 → 新库 status 值
-- pending → pending（保持不变）
-- success → success（保持不变）
```

### Q4: 迁移后 `book_data` 字段显示为字符串而非 JSON

检查 psycopg3 是否正确解析了 jsonb 类型。在新库中查询时使用 `->>` 操作符：

```sql
SELECT book_id, book_data->>'bookName' AS book_name FROM public.books LIMIT 5;
```

### Q5: 迁移后 Web 管理面板不显示新书

新项目的 Web 后端 `books.py` 查询的是 `public.books` 表。迁移后数据已在新库中，刷新页面即可。如果仍不显示，检查数据库连接是否指向正确的库。
